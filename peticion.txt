The current project must now follow the “Battle of Cells” business requirements:

* Local web app.
* Local turn-based simulation.
* Two local players.
* Each player represents a team/species.
* Each player writes a Python-like function that controls the behavior of that player’s cells.
* Each living cell executes one action per turn according to its player’s function.

Main workflow:

1. Show configuration screen.
2. Player 1 enters name and color.
3. Player 1 writes function.
4. Player 1 validates function.
5. If invalid, show error and allow correction.
6. If valid, confirm Player 1.
7. Player 2 enters name and color.
8. Player 2 writes function.
9. Player 2 validates function.
10. If invalid, show error and allow correction.
11. If valid, confirm Player 2.
12. When both players are confirmed, allow starting the simulation.
13. Validate both player functions again before starting.
14. Create a 100x200 board.
15. Create one random initial cell for each player.
16. Initialize turn = 1.
17. Show board, current turn, and statistics.
18. User presses Play.
19. Start the turn cycle.
20. Execute turns until:

    * one team remains alive,
    * both teams die,
    * or the turn limit is reached.
21. Show final result screen.

Important rule change:

Remove the old age-damage and old-age-death rules.

Do not implement this old rule:

* cells lose 5 health from age 70 onward,
* cells die automatically at age 90.

This must be removed from the implementation, tests, and documentation.

Age must still exist, but only for execution priority and display.

Correct age rule for this version:

* Each cell has an age.
* Initial cells start at age 1.
* Newborn cells start at age 1.
* Age increases by 1 at the end of each turn for cells that survived the turn.
* Age does not cause health loss.
* Age does not directly kill cells.
* Age is used for execution priority:

  * lower age acts first,
  * then lower creation turn,
  * then lower row at the start-of-turn snapshot,
  * then lower column at the start-of-turn snapshot.
* Do not use visible unique IDs as business rules for priority.
* Internal IDs may exist only as technical implementation details.

Core board rules:

* The board is fixed at 100 rows x 200 columns.
* The board does not change size during a match.
* No obstacles.
* No resources.
* No external food.
* No more than one cell can occupy the same square.
* Squares outside the board are invalid.
* Initial placement must avoid occupied squares.
* The board must be displayed in the simulation screen.
* The logic must be separated from the UI and must be testable without React.

Cell attributes:

Each cell must have at least:

* team/player reference,
* position,
* health,
* age,
* alive/dead status,
* creation turn,
* last recorded action if available.

Cells do not need a visible unique identifier in the first version.

Health rules:

* Health ranges from 0 to 100.
* Initial cells start with 100 health.
* If health reaches 0, the cell dies.
* Dead cells are immediately removed from the board.
* Health cannot go below 0.
* Resting recovers 3 health.
* Resting cannot increase health above 100.
* Eating/attacking causes 5 damage to the target enemy cell.
* Movement does not cost health.
* Eating does not recover health.

Movement rules:

* A cell can interact only with the 8 neighboring squares.
* Valid directions:

  * n: north
  * s: south
  * e: east
  * w: west
  * ne: northeast
  * nw: northwest
  * se: southeast
  * sw: southwest
* A cell moves one square per turn.
* A cell may only move to an unoccupied square.
* If the destination is outside the board, the action is canceled and the cell loses its action.
* If the destination is occupied by an allied or enemy cell, the action is canceled and the cell loses its action.
* Moving toward an enemy cell is not an attack.
* To damage an enemy, the eat action must be used.

Eat action rules:

* Eat is the attack action.
* The return code uses the letter “a”.
* A cell can only eat enemy cells.
* A cell cannot eat allied cells.
* Eating distance is one square.
* Eating must specify a valid direction.
* If the target square is outside the board, the action is canceled.
* If there is no enemy cell in that direction, the action is canceled.
* If there is an enemy cell, apply 5 damage.
* If the enemy reaches 0 health, remove it immediately from the board.
* Eating does not heal the attacker.

Reproduction rules:

A cell can reproduce only if all conditions are met:

* The cell has at least 50 health before reproducing.
* The cell is less than 55 years old.
* The target direction is valid.
* The target square is inside the board.
* The target square is unoccupied.
* The returned action code is valid.

If reproduction is valid:

* The original cell’s current health is split in half.
* If health is even, both cells receive half.
* If health is odd, the original cell keeps the extra point.
* Example: 51 becomes 26 for original and 25 for newborn.
* Total health must be preserved.
* The newborn belongs to the same team.
* The newborn is placed in the selected direction.
* The newborn starts at age 1.
* The newborn does not act on the same turn in which it is born.
* It can act starting from the following turn.

If reproduction is invalid due to board conditions or health/age conditions:

* The action is canceled.
* The cell loses its action for that turn.

Rest rules:

* Rest consumes the cell’s action.
* Rest restores 3 health.
* Health cannot exceed 100.
* A cell may rest at 100 health, but it gains no extra health.

Turn system:

* The global turn starts at 1.
* Before Play, both players must have:

  * name,
  * color,
  * valid function,
  * confirmed configuration.
* When Play is pressed, the match is locked.
* Once locked, players, colors, code, rules, turn limit, and initial conditions cannot be edited.
* Pause, if present, only pauses visualization or automatic execution.
* Pause must not unlock editing.
* Cells created during a turn do not act in that same turn.
* Actions are resolved one by one.
* The board is updated immediately after each valid action.
* There are no simultaneous actions.
* Each action uses the current board state at the moment the action is executed.
* At the start of each turn, take a snapshot for execution ordering.
* Sort living cells using:

  1. lower age first,
  2. lower creation turn first,
  3. lower row in the start-of-turn snapshot,
  4. lower column in the start-of-turn snapshot.
* If a cell dies before its action arrives, it does not execute.
* At the end of the turn:

  * increase age by 1 for living cells that survived,
  * do not apply aging damage,
  * do not kill cells because of old age,
  * evaluate end conditions,
  * increment turn if simulation continues.

Function programming rules:

Each player writes a Python-like function.

The first version is not full Python.

* direct mutation of game state,
* direct modification of health,
* direct modification of age,

The function must decide only which action the cell attempts to perform.

Victory and end conditions:

* Default turn limit: 5000 turns.
* If the limit is 5000, turn 5000 must be executed fully.
* After the last allowed turn is fully executed, evaluate final result.
* Simulation ends if only one team remains alive.
* Simulation ends if both teams have no living cells.
* Simulation ends if the turn limit is reached.
* If only one team remains alive, that team wins.
* If both teams have no living cells during the same turn close, the result is a draw.
* If the turn limit is reached:

  1. the team with more living cells wins,
  2. if tied, the team with higher total health wins,
  3. if still tied, the match is a draw.
